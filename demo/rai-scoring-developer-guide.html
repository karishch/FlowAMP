🛡️ Responsible AI Scoring — Production Developer Guide

Energy Agent Management Platform · How to build real RAI scoring from AWS signals

Context: The current rai-scorer Lambda runs daily via EventBridge and computes four sub-scores (fairness, transparency, accountability, ethics) per agent. This guide explains how to move from demo placeholders to real production signals using Bedrock Guardrails, CloudWatch, CloudTrail, and AWS Config.

The scorer scans the AgentTable, skips non-agent records (prefixed compliance:, aop:, access:), and computes four equally-weighted sub-scores per agent. The overall score is the weighted average, optionally modified by a system-level NIST RMF compliance factor.

1. Fairness (25% weight)

⚖️
Goal: Detect and prevent biased or unfair outputs from agents.

What the demo does DEMO

Queries an assumed GuardrailInterventions metric that may not exist with the expected dimensions.

What production needs PRODUCTION

  • Bedrock Guardrails — Create a guardrail with content filters and contextual grounding checks, then attach it to your agent. Real CloudWatch metrics are emitted under AWS/Bedrock with dimension GuardrailId.
  • Invocation log sampling — Enable Bedrock model invocation logging to S3. Run a batch job that samples input/output pairs and checks for demographic disparity in responses.
  • Human feedback loop — Store thumbs-up/down from operators. If a specific agent gets disproportionate negative feedback on certain input categories, that's a fairness signal.

Real Signal Sources

CloudWatch: AWS/Bedrock → GuardrailId dimension → Invocations, GuardrailsIntervened S3 logs: Bedrock invocation logs → parse input/output → run bias classifier DynamoDB: User feedback records → aggregate by agent + input category

Scoring Logic

def score_fairness(agent):
    invocations = get_metric('AWS/Bedrock', 'Invocations',
        [{'Name': 'GuardrailId', 'Value': agent['guardrailId']}])
    interventions = get_metric('AWS/Bedrock', 'GuardrailsIntervened',
        [{'Name': 'GuardrailId', 'Value': agent['guardrailId']}])
    intervention_rate = interventions / max(invocations, 1)

    # Also factor in user feedback if available
    feedback = get_feedback_stats(agent['agentId'])
    negative_rate = feedback['negative'] / max(feedback['total'], 1)

    # Base 95, penalize for high intervention rate and negative feedback
    score = 95 - (intervention_rate * 100 * 0.5) - (negative_rate * 100 * 0.3)
    return max(50, min(100, round(score)))

2. Transparency (25% weight)

🔍
Goal: Ensure agent decisions are auditable and traceable.

What the demo does DEMO

Counts CloudTrail events mentioning the agent's resource name. Blunt — doesn't distinguish event types or check if logging is actually enabled.

What production needs PRODUCTION

  • CloudTrail coverage — Verify bedrock:InvokeAgent, bedrock:InvokeModel, and lambda:Invoke events exist for each agent's resources in the last 24h.
  • Bedrock invocation logging — Call GetModelInvocationLoggingConfiguration to check if full input/output logging to S3 or CloudWatch Logs is enabled. This is the real transparency layer.
  • Agent trace visibility — Bedrock Agents emit trace events showing the reasoning chain. Check if enableTrace is set on agent invocations so operators can see why the agent made a decision.
  • CloudWatch Log groups — Verify log groups exist for the agent's Lambda functions (decisions are recorded).

Real Signal Sources

CloudTrail: LookupEvents for bedrock:InvokeAgent, bedrock:InvokeModel Bedrock: GetModelInvocationLoggingConfiguration → is logging enabled? Agent trace: Check if agent alias has trace enabled in invocation config CW Logs: DescribeLogGroups → do log groups exist for agent Lambdas?

Scoring Logic

def score_transparency(agent):
    score = 60  # baseline

    # +15 if CloudTrail has bedrock/lambda events for this agent
    trail_events = count_cloudtrail_events(agent['agentId'])
    if trail_events > 0:
        score += 15

    # +15 if Bedrock invocation logging is enabled
    try:
        logging_config = bedrock.get_model_invocation_logging_configuration()
        config = logging_config.get('loggingConfig', {})
        if config.get('s3Config', {}).get('bucketName') or \
           config.get('cloudWatchConfig', {}).get('logGroupName'):
            score += 15
    except Exception:
        pass  # logging not configured

    # +10 if CloudWatch log groups exist for agent Lambda
    try:
        logs_client = boto3.client('logs')
        result = logs_client.describe_log_groups(
            logGroupNamePrefix=f'/aws/lambda/{agent["functionName"]}')
        if result.get('logGroups'):
            score += 10
    except Exception:
        pass

    return min(100, score)

3. Accountability (25% weight)

📋
Goal: Ensure failures are handled properly and someone is responsible.

What the demo does DEMO

Reads error rate from DynamoDB fields and checks a single Lambda error metric. Doesn't verify DLQs, alarms, or ownership.

What production needs PRODUCTION

  • Lambda error rate — Query AWS/Lambda → Errors and Invocations by FunctionName. This is the primary signal.
  • Dead Letter Queue — Call GetFunctionConfiguration and check DeadLetterConfig exists. No DLQ = failures silently disappear = low accountability.
  • CloudWatch Alarms — Call DescribeAlarms filtered by agent resources. No alarms = nobody is watching.
  • Ownership metadata — Check if the agent record in DynamoDB has owner, team, escalationContact fields populated.
  • Staleness check — An agent untouched for 6+ months with no owner is an accountability risk. Check lastUpdated field.

Real Signal Sources

CloudWatch: AWS/Lambda → Errors, Invocations by FunctionName Lambda: GetFunctionConfiguration → check DeadLetterConfig.TargetArn exists CloudWatch: DescribeAlarms → filter by agent-related resource names DynamoDB: agent record → check owner, team, lastUpdated fields

Scoring Logic

def score_accountability(agent):
    fn_name = agent.get('functionName', 'AgentHandlerFn')
    invocations = get_metric('AWS/Lambda', 'Invocations',
        [{'Name': 'FunctionName', 'Value': fn_name}])
    errors = get_metric('AWS/Lambda', 'Errors',
        [{'Name': 'FunctionName', 'Value': fn_name}])
    error_rate = errors / max(invocations, 1)

    score = 95 - (error_rate * 200)  # 5% error rate → score 85

    # Check DLQ exists
    try:
        fn_config = lambda_client.get_function_configuration(
            FunctionName=fn_name)
        if not fn_config.get('DeadLetterConfig', {}).get('TargetArn'):
            score -= 10  # no DLQ configured
    except Exception:
        score -= 10

    # Check alarms exist for this agent
    alarms = cw.describe_alarms(
        AlarmNamePrefix=agent['agentId'])
    if not alarms.get('MetricAlarms'):
        score -= 5  # no monitoring alarms

    # Check ownership metadata
    if not agent.get('owner'):
        score -= 10  # no owner assigned

    # Check staleness (last updated > 180 days ago)
    last_updated = agent.get('lastUpdated', '')
    if last_updated:
        days_old = (datetime.utcnow() - datetime.fromisoformat(
            last_updated.rstrip('Z'))).days
        if days_old > 180:
            score -= 5  # stale agent

    return max(50, min(100, round(score)))

4. Ethics (25% weight)

🧭
Goal: Ensure agents avoid harmful content and protect PII.

What the demo does DEMO

Queries assumed GuardrailBlocked and GuardrailPiiRedacted metrics that may not exist.

What production needs PRODUCTION

  • Guardrail attachment check — Call bedrock:GetAgent and verify guardrailConfiguration exists. An agent with no guardrail = ethics score drops hard.
  • Guardrail policy verification — Call bedrock:GetGuardrail and check that content filters, PII filters, denied topics, and contextual grounding policies are configured.
  • Guardrail runtime metrics — After attaching a guardrail, CloudWatch emits real metrics under AWS/Bedrock with GuardrailId and GuardrailVersion dimensions.
  • Invocation log sampling — Sample outputs from S3 invocation logs and scan for PII leakage that the guardrail may have missed.

Real Signal Sources

Bedrock: GetAgent → check guardrailConfiguration exists Bedrock: GetGuardrail → verify contentPolicy, sensitiveInformationPolicy, topicPolicy CloudWatch: AWS/Bedrock → GuardrailId dimension → runtime block/intervene metrics S3 logs: Invocation logs → sample outputs for PII leakage

Scoring Logic

def score_ethics(agent):
    score = 60  # baseline without guardrail

    # Check if guardrail is attached to the agent
    try:
        agent_config = bedrock.get_agent(agentId=agent['agentId'])
        guardrail = agent_config.get('agent', {}).get(
            'guardrailConfiguration')
    except Exception:
        guardrail = None

    if guardrail:
        score += 20  # guardrail exists and is attached

        # Check what policies the guardrail has
        try:
            gr = bedrock.get_guardrail(
                guardrailIdentifier=guardrail['guardrailIdentifier'])
            if gr.get('contentPolicy'):
                score += 5   # content filters active
            if gr.get('sensitiveInformationPolicy'):
                score += 10  # PII protection active
            if gr.get('topicPolicy'):
                score += 5   # denied topics configured
        except Exception:
            pass

    return min(100, score)

5. System-Level: NIST RMF Modifier

🏛️
Goal: Weak system-level risk posture should drag down all agent scores.

The current implementation reads a compliance:nist-sp800-37 record from DynamoDB. In production, tie this to real AWS security posture:

Option A: AWS Security Hub

def get_rmf_modifier():
    sh = boto3.client('securityhub')
    # Get the overall security score from Security Hub
    response = sh.get_security_control_definitions()
    # Parse compliance percentage from findings
    findings = sh.get_findings(Filters={
        'ComplianceStatus': [{'Value': 'FAILED', 'Comparison': 'EQUALS'}],
        'RecordState': [{'Value': 'ACTIVE', 'Comparison': 'EQUALS'}]
    })
    failed_count = len(findings.get('Findings', []))
    # More failures = lower modifier (0.90 to 1.0 range)
    return max(0.90, 1.0 - (failed_count * 0.002))

Option B: AWS Config Rules

def get_rmf_modifier():
    config = boto3.client('config')
    rules = config.describe_compliance_by_config_rule(
        ComplianceTypes=['COMPLIANT', 'NON_COMPLIANT']
    )['ComplianceByConfigRules']
    compliant = sum(1 for r in rules
        if r['Compliance']['ComplianceType'] == 'COMPLIANT')
    total = len(rules) or 1
    ratio = compliant / total
    # 100% compliant → 1.0, 80% → 0.96, 60% → 0.92
    return max(0.90, 0.80 + (ratio * 0.20))

The modifier is applied as: overall_score = round(weighted_average * rmf_modifier)

To make the scorer work with real signals, add these to energy-platform-cdk-stack.ts:

Additional IAM Permissions for RAI Scorer Lambda

raiScorerFn.addToRolePolicy(new iam.PolicyStatement({
  actions: [
    // Existing
    'cloudwatch:GetMetricStatistics',
    'cloudwatch:ListMetrics',
    'cloudtrail:LookupEvents',
    // New for production
    'bedrock:GetAgent',
    'bedrock:GetGuardrail',
    'bedrock:GetModelInvocationLoggingConfiguration',
    'lambda:GetFunctionConfiguration',
    'cloudwatch:DescribeAlarms',
    'logs:DescribeLogGroups',
    // If using Security Hub
    'securityhub:GetFindings',
    // If using AWS Config
    'config:DescribeComplianceByConfigRule',
  ],
  resources: ['*'],
}));

Add a Bedrock Guardrail (required for ethics + fairness scoring)

const guardrail = new bedrock.Guardrail(this, 'EnergyGuardrail', {
  name: 'energy-agent-guardrail',
  blockedInputMessaging: 'Request blocked by content policy.',
  blockedOutputsMessaging: 'Response blocked by content policy.',
  contentFilters: [
    {
      type: bedrock.ContentFilterType.HATE,
      inputStrength: bedrock.FilterStrength.HIGH,
      outputStrength: bedrock.FilterStrength.HIGH,
    },
    {
      type: bedrock.ContentFilterType.VIOLENCE,
      inputStrength: bedrock.FilterStrength.HIGH,
      outputStrength: bedrock.FilterStrength.HIGH,
    },
  ],
  piiFilters: [
    {
      type: bedrock.PiiEntityType.EMAIL,
      action: bedrock.GuardrailAction.ANONYMIZE,
    },
    {
      type: bedrock.PiiEntityType.PHONE,
      action: bedrock.GuardrailAction.ANONYMIZE,
    },
  ],
});

// Attach guardrail to the agent
agent.addGuardrail(guardrail);

Enable Bedrock Invocation Logging (required for transparency scoring)

// Create S3 bucket for invocation logs
const invocationLogBucket = new s3.Bucket(this, 'InvocationLogBucket', {
  removalPolicy: cdk.RemovalPolicy.RETAIN,
  encryption: s3.BucketEncryption.S3_MANAGED,
  lifecycleRules: [{ expiration: cdk.Duration.days(90) }],

// Enable via custom resource (no L2 construct yet)
new cr.AwsCustomResource(this, 'EnableInvocationLogging', {
  onCreate: {
    service: 'Bedrock',
    action: 'putModelInvocationLoggingConfiguration',
    parameters: {
      loggingConfig: {
        s3Config: {
          bucketName: invocationLogBucket.bucketName,
          keyPrefix: 'invocation-logs/',
        },
        embeddingDataDeliveryEnabled: false,
        imageDataDeliveryEnabled: false,
        textDataDeliveryEnabled: true,
      },
    },
    physicalResourceId: cr.PhysicalResourceId.of('invocation-logging'),
  },
  policy: cr.AwsCustomResourcePolicy.fromStatements([
    new iam.PolicyStatement({
      actions: ['bedrock:PutModelInvocationLoggingConfiguration'],
      resources: ['*'],
    }),
  ]),
});

7. Summary: Demo → Production Gap

Pillar Demo (Current) Production (Real Signals)
Fairness Assumed CW metric that may not exist Guardrail intervention rate + invocation log sampling + user feedback
Transparency CloudTrail event count (blunt) Invocation logging enabled? + Trace enabled? + Log groups exist?
Accountability Error rate from DDB fields Lambda error rate + DLQ exists? + Alarms exist? + Owner assigned?
Ethics Assumed guardrail block metrics Guardrail attached? + Content/PII/topic policies configured?
System-Level DDB compliance record Security Hub score or AWS Config rule compliance
💡 Key Insight: In production, the scorer should check configuration (is the guardrail attached? is logging enabled? is there a DLQ?) as much as runtime metrics. A well-configured agent with no incidents scores higher than a poorly-configured agent that hasn't been caught yet.
